# Agentic & Adaptive RAG

So far, we've seen RAG pipelines that follow a fixed path: receive question, retrieve documents, generate answer. But what if the system could **decide on its own** which strategy to use, evaluate whether the answer is good enough, and **try again** if it isn't? That's the leap from traditional RAG to **Agentic & Adaptive RAG**: pipelines that think, plan, and self-correct.

---

## 1. Adaptive RAG — Choosing the right strategy

Adaptive RAG starts from a simple premise: **not every question needs the same treatment**. Before retrieving anything, a classifier analyzes the query and decides which route to follow.

### How does it work?

```
User Query
    │
    ▼
[ Query Classifier ]
    │
    ├── Simple ──────► Direct index search
    │
    ├── Complex ─────► Decomposition into sub-questions
    │                  (each with its own retrieval)
    │
    └── Out of scope ► Direct LLM response
                       (no retrieval)
```

### Routing types

| Query Type | Strategy | Example |
| :--- | :--- | :--- |
| Simple factual | Direct search | _"What is GPT-4's token limit?"_ |
| Multi-faceted | Decomposition + multiple searches | _"Compare the pros and cons of PostgreSQL vs MongoDB for an e-commerce system"_ |
| Conversational | Reformulation with chat context | _"What about the pricing?"_ (referring to the previous topic) |
| Out of scope | No retrieval, LLM response | _"What is 2 + 2?"_ |

The classifier can be an LLM with a specific prompt, a model trained for classification, or even simple heuristic rules. The key point is that the **route is decided dynamically**, not hardcoded.

## 2. Agentic Loop — Planning & Reflection

This is where RAG gains true autonomy. Instead of a linear pipeline, the system operates in an **iterative loop** with three fundamental capabilities:

### 2.1 Planning

The agent receives a complex task and breaks it down into steps before executing anything.

```
Question: "Which Brazilian AI companies received Series B investment
           in 2024 and what was the market impact?"

Plan generated by the agent:
  1. Search for Brazilian AI companies with Series B funding in 2024
  2. For each company found, retrieve investment details
  3. Search for market analyses about AI in Brazil in 2024
  4. Synthesize everything into a coherent answer
```

Each step of the plan is executed as an independent retrieval action and can generate new steps as the agent discovers more information.

### 2.2 Reflection

After generating a response (or an intermediate result), the agent **evaluates its own output**:

```
┌─────────────────────────────────────────────┐
│              Agentic Loop                   │
│                                             │
│  ┌─────────┐    ┌──────────┐    ┌────────┐ │
│  │  Plan   │───►│ Execute  │───►│Reflect │ │
│  └────▲────┘    └──────────┘    └───┬────┘ │
│       │                             │      │
│       │    ┌──────────────┐         │      │
│       └────┤ Needs        │◄────────┘      │
│            │ improvement? │                │
│            └──────┬───────┘                │
│                   │ No                     │
│                   ▼                        │
│           [ Final Answer ]                 │
└─────────────────────────────────────────────┘
```

The questions the agent asks itself during reflection:

- **Relevance:** Are the retrieved documents relevant to the question?
- **Completeness:** Does the answer cover all aspects of the question?
- **Faithfulness:** Is the answer faithful to the source documents (no hallucination)?
- **Coherence:** Is the answer internally consistent and well-structured?

If any criterion fails, the agent decides on the corrective action:

| Detected Problem | Corrective Action |
| :--- | :--- |
| Irrelevant documents | Reformulate the query and search again |
| Incomplete answer | Identify gaps and perform additional searches |
| Hallucination detected | Discard the passage and regenerate based on documents |
| Ambiguous query | Request clarification or test multiple interpretations |

### 2.3 Tool Use

The agent is not limited to searching a single index. It can have access to multiple tools:

- **Vector search** on the internal knowledge base
- **Web search** for recent information
- **SQL** to query structured databases
- **APIs** for real-time data (prices, weather, status)
- **Calculator** for mathematical operations

The agent decides **which tool to use** at each step of the plan — hence the name "agentic."

## 3. Frameworks and Patterns

### CRAG (Corrective RAG)

CRAG adds a **relevance evaluation** step right after retrieval:

```
Query → Retrieval → [Evaluate Relevance] → Correct? ─── Yes ──► Generate Answer
                                              │
                                             No
                                              │
                                              ▼
                                     Supplementary Web Search
                                              │
                                              ▼
                                       Generate Answer
```

If the retrieved documents don't pass the relevance filter, CRAG falls back to a web search — combining the best of internal knowledge with external information.

### Self-RAG

Self-RAG trains the LLM to emit **special reflection tokens** alongside the response:

- `[Retrieve]` — "I need to search for more information to answer this"
- `[IsRel]` — "Is this document relevant to the question?"
- `[IsSup]` — "Is my statement supported by the document?"
- `[IsUse]` — "Is my answer useful to the user?"

This turns reflection into part of the generation process, without requiring extra LLM calls.

## 4. Concrete Example — Technical Support Agent

Imagine a support agent for a SaaS platform:

**User question:** _"My Stripe integration stopped working after yesterday's update"_

**Iteration 1 — Planning:**
1. Search the changelog for the most recent update
2. Search the Stripe integration documentation
3. Search for known issues related to Stripe

**Iteration 1 — Execution:**
- Retrieves changelog → finds: _"Authentication headers updated for API v3"_
- Retrieves Stripe docs → finds: _"The integration uses the `X-Auth-Token` header"_
- Searches issues → no open issues

**Iteration 1 — Reflection:**
- _"The changelog mentions a change in authentication headers. The Stripe integration uses an auth header. There's a probable correlation, but I need to verify whether the specific header was affected."_

**Iteration 2 — Additional execution:**
- Specific search: _"X-Auth-Token changes API v3"_
- Finds: _"Header `X-Auth-Token` renamed to `Authorization: Bearer` in v3"_

**Final answer:**
_"Yesterday's update migrated the API to v3, which renames the authentication header from `X-Auth-Token` to `Authorization: Bearer`. The Stripe integration needs to be updated to use the new header format."_

Without the agentic loop, the system would have returned only the generic changelog without making the connection between the change and the specific problem.

## 5. Advantages and Limitations

**More accurate answers:** The reflection loop catches and corrects errors before delivering to the user.

**Handles complex questions:** Decomposition and planning enable answering questions that require multiple sources.

**Self-correction:** The system detects hallucinations and irrelevant documents without human intervention.

**Flexibility:** Different strategies for different query types, instead of one-size-fits-all.

**Higher latency:** Multiple iterations mean more LLM calls and longer response times.

**Cost:** Each loop iteration consumes tokens — a single question can generate 3-5x more calls than a simple RAG.

**Risk of infinite loops:** Without well-defined limits, the agent can get stuck refining indefinitely. It's essential to define a **maximum number of iterations** and clear stopping criteria.

**Implementation complexity:** Orchestrating planning, execution, and reflection requires robust frameworks (LangGraph, CrewAI, Autogen).

---

## Conclusion

Agentic & Adaptive RAG represents the evolution of retrieval from a **passive pipeline** to an **autonomous system**. Adaptive RAG brings intelligence to strategy selection, while the agentic loop with planning and reflection allows the system to reason about its own limitations and self-correct. The cost is higher latency and complexity, but the quality gain — especially for complex questions — is transformative. Frameworks like CRAG and Self-RAG show that this evolution is already well-established in research and rapidly making its way into production.
